# Lesson 2 — Vec2

> **Goal**: Replace the two loose `x` and `y` floats with a proper 2D vector type.
>
> **Concepts**: `struct`, named fields, `impl`, constructor methods, field shorthand, `#[derive(Clone, Copy)]`.

---

## The problem with loose floats

Right now your position is two separate variables:

```rust
let x = 400.0_f32;
let y = 300.0_f32;
```

That works for one value, but a physics engine needs to add positions, subtract them, scale them. Doing that with loose pairs of floats means every operation is two lines instead of one. Mistakes creep in. It gets unwieldy fast.

What we want is to treat a 2D point as a single thing — a **vector**.

---

## Defining a struct

A `struct` groups related values under one name:

```rust
struct Vec2 {
    x: f32,
    y: f32,
}
```

`x` and `y` are **fields**. Each field has a name and a type. `f32` is a 32-bit floating-point number — the standard choice for game math.

You can now write:

```rust
let position = Vec2 { x: 400.0, y: 300.0 };
```

And access the fields with `.`:

```rust
draw_circle(position.x, position.y, 20.0, WHITE);
```

---

## Adding a constructor with `impl`

Creating a `Vec2` by writing `Vec2 { x: ..., y: ... }` every time is verbose. We can add a `new` method to make it shorter:

```rust
impl Vec2 {
    fn new(x: f32, y: f32) -> Self {
        Vec2 { x, y }
    }
}
```

A few things to notice:

**`impl Vec2`** — opens a block of methods that belong to `Vec2`.

**`fn new(x: f32, y: f32) -> Self`** — a function that takes two floats and returns a `Vec2`. `Self` is an alias for the type you're implementing — here it means `Vec2`.

**`Vec2 { x, y }`** — when a field name matches a local variable name, you can write just the name once. This is called **field shorthand**. It's identical to writing `Vec2 { x: x, y: y }`.

Now you can write:

```rust
let position = Vec2::new(400.0, 300.0);
```

The `::` syntax calls an **associated function** — a function attached to the type rather than to an instance. `new` is the conventional name for constructors in Rust.

---

## `#[derive(Clone, Copy)]`

Add this line above your struct:

```rust
#[derive(Clone, Copy)]
struct Vec2 {
    x: f32,
    y: f32,
}
```

`#[derive(...)]` is an attribute that auto-generates code. Here it generates two **trait** implementations — `Clone` and `Copy`. A trait is a contract: it says "this type can do X." Let's look at each one.

---

### `Clone` — explicit duplication

`Clone` adds a `.clone()` method that produces an independent copy of the value:

```rust
let a = Vec2::new(1.0, 2.0);
let b = a.clone();   // b is a fresh copy; a is untouched
```

Cloning is always **explicit** — Rust will never clone something behind your back. This matters for expensive types like `Vec<T>` or `String`, where cloning allocates new heap memory. For those types, you want to see every clone in the source code.

`#[derive(Clone)]` generates the implementation automatically: it clones each field. For `Vec2`, that means copying two `f32` values — essentially free.

---

### `Copy` — implicit duplication

`Copy` changes the default behaviour of assignment and function calls:

| Without `Copy` | With `Copy` |
|----------------|-------------|
| `let b = a;` moves `a` into `b` — `a` is gone | `let b = a;` copies `a` into `b` — both are valid |
| Passing `a` to a function moves it — you can't use `a` afterwards | Passing `a` to a function copies it — `a` still works |

```rust
// Without Copy — move semantics:
let a = Vec2::new(1.0, 2.0);
let b = a;
draw_circle(a.x, a.y, 10.0, WHITE);  // ERROR: a was moved

// With Copy — copy semantics:
let a = Vec2::new(1.0, 2.0);
let b = a;
draw_circle(a.x, a.y, 10.0, WHITE);  // fine — a was copied
```

`Copy` is a **bitwise copy** — the raw bytes are duplicated. This is only valid when a type doesn't own heap memory. `f32` owns nothing; two `f32` fields own nothing; so `Vec2` can be `Copy`. A type like `String` cannot be `Copy` — it points to heap-allocated memory, and a bitwise copy would give you two owners of the same allocation, which would double-free on drop.

---

### Why do we need both?

`Copy` is a **subtrait** of `Clone` — it extends it. Rust requires that any `Copy` type also implements `Clone`. The reasoning: if something can be copied cheaply and implicitly, it can certainly be cloned explicitly too.

If you try to derive `Copy` without `Clone`, the compiler rejects it:

```
error: the trait `Clone` is not implemented for `Vec2`
```

So the two always appear together for types like `Vec2`.

---

### When should you add `Copy`?

Ask: is this type small, cheap to duplicate, and free of heap allocations?

| Type | `Copy`? | Why |
|------|---------|-----|
| `f32`, `i32`, `bool` | yes | primitive, bitwise copy is correct |
| `Vec2` (two `f32`s) | yes | small, no heap, copying is cheap |
| `Body` (our ball struct) | no | will grow large; one clear owner makes sense |
| `String` | no | owns heap memory; copy would double-free |
| `Vec<T>` | no | same reason as `String` |

For `Vec2`, `Copy` is the right call. Every physics calculation — `position + velocity * dt` — hands `Vec2` values into operators and functions. Without `Copy`, each of those would consume the original, forcing awkward cloning everywhere. With `Copy`, the math reads naturally.
